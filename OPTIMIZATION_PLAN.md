# Barley Grid 优化方案

## 背景

Barley Grid 当前是一个 Vite + React 的 F1 数据看板，已有完整的数据展示、赛程、积分榜、分站详情抽屉和车手/车队详情。现在主要目标不是重做功能，而是把体验升级成新版设计方向：

- 更快看到核心数据
- 更稳定的数据更新
- 更鲜活、有 F1 车手网站气质的视觉系统
- 更简洁的首页信息架构
- 更好的移动端和深链接体验

新版视觉参考方向：黑白高反差骨架、F1 红主色、少量荧光黄绿与亮青作为速度/数据状态点缀。整体接近现代 F1 车手个人网站 + 轻量数据看板，而不是传统灰白 SaaS 卡片页。

## 当前主要问题

### 1. 首屏加载被人为延迟

`src/App.jsx` 中首次加载设置了至少 1.8 秒的展示延迟。即使接口已经返回，用户也必须等待 loading 动画结束。

影响：

- 用户体感数据更新慢
- 快速网络下也无法立即看到页面
- loading 变成阻塞体验，而不是缓冲体验

### 2. 首页数据请求过重

`fetchAllData()` 首次加载并发请求 5 个外部接口：

- driver standings
- constructor standings
- calendar
- race results
- sprint results

这些请求目前被视为一个整体，最慢的接口会拖住整个首页。

影响：

- 首页不能渐进显示
- 任一接口失败时整体容错较弱
- 刷新页面时重复等待完整数据链路

### 3. 主数据缺少持久化缓存

当前主数据没有 localStorage / stale cache。每次刷新页面都会重新请求外部 API。

影响：

- 二次访问体验没有加速
- 外部 API 慢或波动时页面空白更久
- 离线或弱网没有降级数据

### 4. 更新策略太粗

当前每 5 分钟全量轮询一次 `fetchAllData()`。

影响：

- 平时请求过多
- 比赛日又不够细
- 赛程、积分、结果这些数据更新频率不同，却被同一种策略处理

### 5. LiveTiming 代理缓存偏长

`api/f1proxy.js` 和 `api/f1timing/[...path].js` 当前设置 `s-maxage=300`。练习赛、排位赛或刚结束的数据可能被缓存 5 分钟。

影响：

- 用户会感觉 FP/赛后数据更新慢
- 比赛窗口期不够实时

### 6. 分站详情存在二次等待

打开 `RaceDrawer` 后才请求排位、冲刺、练习赛数据。

影响：

- 抽屉打开后可能继续 loading
- 重复打开同一分站时体验不够顺滑

### 7. 新闻模块依赖第三方 RSS 转换服务

`AIInsightsCard` 直接请求 `api.rss2json.com`。

影响：

- 可能限流或失败
- 请求速度不可控
- 新闻模块不是核心体验，却可能影响整体感受

### 8. 深链接被重置

应用初始化时强制把 history state 重置到首页 `/`。

影响：

- 直接访问 `/schedule`、`/standings` 不能稳定进入对应页面
- 分享链接体验不好
- 浏览器前进/后退逻辑复杂

### 9. 移动端布局有溢出风险

倒计时四列、大日期、大卡片和固定间距在窄屏下容易裁切。

影响：

- 手机端首屏不够精致
- 关键数据可能被截断
- 视觉质量下降明显

### 10. 旧视觉系统和新版方向不一致

当前 `.apple-card` 偏大圆角、柔阴影、灰白苹果风。新版方向需要更强的赛车数据感和更鲜活的配色。

影响：

- 只改局部颜色会显得不统一
- 页面缺少记忆点
- F1 气质不够强

## 优化目标

### 性能目标

- 首屏不再有硬性 1.8 秒等待
- 有缓存时优先显示旧数据，再后台刷新
- 首页核心信息先显示，非核心数据异步补齐
- 弱网或接口失败时仍能展示最近一次有效数据

### 数据目标

- 区分数据类型的更新频率
- 赛程/积分长缓存
- 比赛日、下一站、分站结果短缓存
- LiveTiming 在比赛窗口期降低缓存时间
- 抽屉数据按需加载后缓存

### 视觉目标

- 建立新版设计 token
- 降低卡片圆角和阴影
- 使用黑白红作为基础视觉骨架
- 用荧光黄绿标记 active/current/live
- 用亮青标记 timing/data line
- 首页更简洁，更像个人 F1 数据中心

### 产品体验目标

- 首页第一屏只表达 3 个重点：下一站、赛季进度、积分概览
- 下面只保留 3 个轻模块：最近结果、接下来赛程、积分预览
- `/schedule` 和 `/standings` 可直接访问
- 移动端无横向滚动和裁切

## 分阶段执行计划

### Phase 1: 数据与加载体验

优先级最高。

任务：

- 移除首次加载 1.8 秒硬等待
- 增加主数据 localStorage 缓存
- 页面启动时先读取缓存并立即渲染
- 后台请求新数据，成功后静默刷新 UI
- 为 fetch 请求增加超时和局部失败处理
- 拆分 `fetchAllData()`，允许核心数据先返回

建议实现：

- `getCachedData()`
- `setCachedData()`
- `fetchCoreData()`
- `fetchResultsData()`
- `refreshDataInBackground()`

验收标准：

- 二次打开页面几乎立即显示内容
- 外部 API 慢时不会长时间空白
- 接口失败时能展示缓存和错误提示
- `npm run build` 通过

### Phase 2: 数据更新策略

任务：

- 为不同数据设置不同缓存 TTL
- 比赛日或下一场 24 小时内缩短刷新间隔
- 非比赛期降低轮询频率
- RaceDrawer 请求结果后缓存到内存或 localStorage
- LiveTiming 代理按路径设置更合理的 cache-control

建议 TTL：

- 赛程：6-24 小时
- 积分榜：10-30 分钟
- 分站结果：5-10 分钟
- LiveTiming session 数据：30-120 秒，比赛窗口期更短
- 已结束很久的历史 session：长期缓存

验收标准：

- 比赛窗口期数据更及时
- 非比赛期请求明显减少
- 重复打开同一分站不重复等待

### Phase 3: 新视觉系统

任务：

- 重写全局色彩 token
- 重构 `.apple-card` 为新版 card 样式
- 定义新版按钮、chip、tab、section heading
- 去掉大面积柔和渐变和装饰光斑
- 建立 F1 风格元素：diagonal cut、track line、timing tick、team strip

建议配色比例：

- 70% warm white / graphite / neutral
- 20% racing red
- 7% chartreuse-lime
- 3% cyan timing accent

建议 token：

```css
--bg: #f7f4ee;
--surface: #ffffff;
--ink: #101010;
--muted: #77746f;
--line: rgba(16, 16, 16, 0.1);
--red: #ff2d2d;
--lime: #d7ff38;
--cyan: #20d7ff;
--graphite: #171717;
```

验收标准：

- 页面整体摆脱旧 Apple 卡片风
- 配色更鲜活但不杂乱
- 组件样式统一
- 移动端文字和按钮不溢出

### Phase 4: 首页重构

任务：

- 用新版信息架构替换当前首页卡片堆叠
- 首屏做成一个简洁 dashboard band
- 合并下一站、赛季进度、Top 3 standings
- 下方保留 3 个模块：最近结果、赛程预览、积分预览
- 降低页面纵向冗余

新版首页结构：

1. Header
2. Hero dashboard band
3. Recent result
4. Upcoming schedule
5. Standings preview
6. Footer

验收标准：

- 首屏不复杂，但有明显 F1 视觉记忆点
- 用户 3 秒内知道下一站和赛季状态
- 信息密度比当前更清爽
- 桌面和移动端都无明显裁切

### Phase 5: 路由和移动端

任务：

- 根据当前 URL 初始化 `currentView`
- 不再强制 `replaceState` 到 `/`
- 修复 `/schedule`、`/standings` 深链接
- 处理浏览器前进/后退
- 优化移动端倒计时布局
- 检查所有固定宽度和大字号

验收标准：

- 直接访问 `/schedule` 显示赛程页
- 直接访问 `/standings` 显示积分页
- 手机端无横向滚动
- 主要卡片在 390px 宽度下显示正常

### Phase 6: 可选增强

这些可以晚点做。

- 新闻模块改为自己的 API route + 缓存
- 增加“最近更新于”提示
- 增加手动刷新按钮
- 增加数据来源状态提示
- 增加 skeleton，而不是整页 loading
- 对 driver 图片做 lazy loading
- 清理乱码注释和旧测试文件

## 建议最先做的改动清单

第一轮可以只做这些，风险小、收益明显：

1. 删除首次加载 1.8 秒硬等待
2. 加主数据 localStorage 缓存
3. 有缓存时先显示缓存，再后台刷新
4. 修复深链接初始化
5. 新增基础配色 token
6. 先重做 Header + 首页 hero band

## 风险点

- 外部 F1 API 的速度和稳定性不可控，需要缓存兜底
- LiveTiming 数据路径和 session 命名可能变化，需要保守处理
- 中文文案文件当前存在明显编码乱码，重构时要统一保存为 UTF-8
- 视觉系统改动会影响多个组件，建议先建 token，再逐个组件替换
- 不要一次性重写所有页面，否则难以判断问题来自数据还是 UI

## 验证方式

每个阶段完成后至少执行：

```bash
npm run build
```

视觉改动阶段建议检查：

- desktop: 1440px
- tablet: 768px
- mobile: 390px

重点检查：

- 首屏是否快速显示
- 移动端是否横向溢出
- 倒计时是否裁切
- 颜色是否过度花哨
- `/schedule` 和 `/standings` 是否可直接访问

## 推荐执行顺序

```text
数据加载体验 -> 数据更新策略 -> 设计 token -> 首页重构 -> 路由/移动端 -> 可选增强
```

这样可以先解决“慢”的体感，再把新版设计稳稳落到代码里。
